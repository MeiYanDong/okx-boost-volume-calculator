# OKX Boost 钱包总览工具

这个项目用来帮你计算 BNB Chain 和 X Layer 上的 OKX Boost 交易量。

它不是给技术同学看的区块链脚本，而是给正在参加 OKX Boost 活动的人用的工作台：你把钱包地址、快照日期和代币加成规则告诉 Agent，Agent 把项目跑起来，页面会帮你看到所有钱包最近 10 天的 Boost 情况。

## 需求

参加 OKX Boost 时，你真正关心的不是“我今天随便刷了几笔”，而是：

1. 每个钱包过去 10 天的 Boost 交易量是多少。
2. 今天有没有继续刷到位。
3. 哪些钱包已经够，哪些钱包还差。
4. 某个代币在某一天有没有额外加成。
5. 新交易出现后，能不能只刷新新增部分。

这些信息如果靠 OKX 钱包页面和区块浏览器手工对，会非常慢，也容易漏掉。

## 痛点

OKX Boost 的计算对新手不友好，主要有几个坑：

1. 统计窗口是“快照日期往前 10 天”，而且快照日期当天包含在内。
2. 今天还没结束时，结果只能是实时预估，后面继续交易会改变结果。
3. 一个钱包可能有很多交易，但只有 BNB Chain / X Layer 上打到 OKX DEX Router 的交易才属于当前工具统计范围。
4. 同一个代币，不同日期可能有不同加成；不同钱包刷同一个代币，也应该分别设置。
5. 钱包多了以后，如果每次都重新扫完整 10 天，会很慢，也浪费 RPC / Ankr 额度。

## 解放方案

这个工具把工作拆成两层：

1. **交易发现**
   优先用 Ankr 钱包索引快速找到 BNB Chain / X Layer 的 OKX 交易 hash；BNB Chain 如果 Ankr 不可用，再尝试 Explorer 和 Chainstack/RPC 兜底。X Layer 不用公共 RPC 扫日志兜底，因为公共节点通常限制 `eth_getLogs` 区间，不适合扫 10 天窗口。

2. **交易解析**
   用对应链的 RPC 读取交易、receipt、token 信息，计算基础倍数、成交额和 Boost 交易量。

当前工作台支持：

- 首页直接看所有钱包的 10 日 Boost 总览
- 最近 10 天数据展示
- 本地归档，扫过的钱包不会重复全量扫描
- 服务端归档，Vercel 重新部署后仍能恢复钱包和结果
- 数据空间隔离，不同用户不会互相覆盖钱包和结果
- 邀请制账号登录，登录后按 Supabase 账号保存云端归档
- 管理员可在偏好设置里生成、查看和撤销邀请码；首个管理员用私有访问码初始化，后续使用管理员账号管理
- 管理员可查看用户、调整钱包额度、启用或禁用账号
- 钱包额度服务端强制生效，普通用户超额后不能继续同步归档
- 登录用户可以在偏好设置里保存自己的飞书机器人配置
- 单钱包增量刷新，只补扫新区块
- 强制重扫，用于重建异常归档
- 批量钱包 3 路有限并行刷新
- 每个钱包、每个日期、每个代币单独设置额外加成
- 快照预警支持一键发送账号级飞书群机器人提醒
- Vercel 私人部署支持每天自动增量刷新，并在未来快照低于目标时按用户配置自动飞书提醒
- 交易发现来源展示：`Ankr 索引`、`Explorer 索引`、`RPC 兜底`、`多链合并`、`本地归档`

## 结果

一次计算后，你应该能看到：

1. 所有钱包的 10 日合计 Boost
2. 所有钱包的 10 日平均 Boost
3. 每个钱包今天的 Boost
4. 每个钱包距离目标线还差多少
5. 每日数据
6. 代币额外加成
7. 每笔交易明细
8. 警告信息和数据来源
9. 未来 3 次快照是否会因为交易量到期而低于目标

如果你设置了目标线，页面会帮你判断整体完成度。

## 你需要准备什么

你只需要准备：

1. 钱包地址，可以是一个，也可以是一批。
2. 快照日期。
3. 目标线，如果你知道。
4. 哪些代币在哪些日期有额外加成，如果你知道。

不知道加成也没关系。可以先跑一次，让页面列出交易里出现过的日期和代币，再逐项填写。

## 如何让 Agent 私人部署

如果你希望部署到 Vercel，直接告诉 Agent：

```text
请把这个项目部署成 Vercel 私人可用版。

请先阅读 docs/vercel-private-deploy.md，然后完成：
1. 检查 Vercel 项目是否已关联正确仓库。
2. 在 Vercel 环境变量里配置 Ankr、BNB Chain RPC、X Layer RPC、访问密码等服务端变量。
3. 配置 Vercel Upstash Redis，用来保存钱包列表、扫描结果、加成规则和自动刷新状态。
4. 如果我要多用户账号隔离，请配置 Supabase：SUPABASE_URL、SUPABASE_PUBLISHABLE_KEY、SUPABASE_SECRET_KEY，并实际验证 /api/auth?action=me 返回 configured: true。
5. 配置 CRON_SECRET，用来保护每日自动刷新接口。
6. 如果我要飞书提醒，请让我登录后在偏好设置里配置个人 Webhook；不要使用全局飞书 Webhook。
7. 部署生产版本。
8. 验证页面能打开。
9. 验证不带访问密码时 API 会拒绝。
10. 验证带访问密码时 Ankr 钱包索引和 RPC 能正常返回。
11. 验证 Supabase 邀请码创建、注册、登录、刷新页面恢复归档。
12. 验证登录用户不填写私有访问码也能扫描，且钱包数量超过账号额度时会被服务端拒绝保存。
13. 验证账号级飞书配置可以保存，且返回给前端的是脱敏信息。
14. 验证 /api/cron/daily-refresh?dryRun=1 能返回自动刷新预览，但不真实发送飞书。
15. 告诉我最终链接、是否启用访问保护、是否完成线上烟测。
```

私人部署不要公开发给不可信的人。否则别人会消耗你的 Ankr、Chainstack 或 Explorer 额度。

## 邀请制怎么管理用户

私人部署建议采用邀请制，不开放公开注册。

管理员进入页面后，在“偏好设置”里填写私有访问码，然后可以：

1. 第一次创建自己的管理员邀请码。
2. 用管理员邀请码注册并登录自己的账号。
3. 后续用管理员账号创建普通用户或管理员邀请码。
4. 给邀请码绑定邮箱，也可以留空。
5. 设置这个用户最多可保存的钱包数量。这个额度会在服务端保存归档时强制校验，不只是页面提示。
6. 查看最近邀请码的状态。
7. 撤销尚未使用的邀请码。

新邀请码原文只会在创建成功后展示一次。后续列表只展示脱敏后的邀请码记录，不展示原文。

只要系统里已经存在 active 管理员，私有访问码就不再允许直接管理邀请码。这样可以避免“知道部署密码的人都能管用户”的问题。

管理员登录后，还可以在“用户管理”里查看已注册用户，调整用户的钱包额度，或者临时禁用账号。禁用后，该账号不能继续扫描或同步云端归档。系统会阻止管理员禁用自己，也会阻止禁用最后一个 active 管理员。

## 快照日期怎么理解

快照日期会被包含在统计窗口里。

例如选择 `2026-05-07`，统计窗口就是包含 `2026-05-07` 在内的最近 10 个 UTC 日期。

如果当天还没结束，页面会提示“实时预估”。这不是错误，只是说明今天后面如果继续交易，结果还会变化。

## 刷新和重扫有什么区别

**刷新**：

- 用于日常更新。
- 如果本地归档已经覆盖最新区块，会直接使用归档。
- 如果有新区块，只补扫新增区块。
- 不会清空当前钱包详情。

**重扫**：

- 用于结果异常、规则变化较大、或者你想重建归档时。
- 会重新扫描整个 10 天窗口。
- 比刷新慢，也更消耗额度。

## 自动归档和定时提醒怎么理解

本地运行时，浏览器会保存一份本地归档；私人部署到 Vercel 后，推荐用 Supabase 账号保存云端归档。旧版 Upstash Redis 数据空间仍然兼容。

这样更新代码或重新部署 Vercel 后，生产链接不变，数据也不会因为重新部署而消失。

线上每天北京时间 08:05 会自动做一次增量刷新。它确认的是上一 UTC 日快照，并同时预测未来 3 次快照。如果某个钱包未来会低于单钱包 10 日累计目标，系统会自动发飞书提醒。

自动提醒只在“有风险”或“自动刷新失败”时发送，不发送无意义的安全日报。

登录用户可以在“偏好设置 → 飞书通知”里保存自己的飞书机器人 Webhook 和签名密钥。手动风险提醒和每日自动提醒只使用这个账号级配置；如果账号没有配置，系统不会代发到其他人的机器人。

“发送真实数据测试”会读取当前登录账号的云端归档，并执行一次每日自动刷新演练，然后把真实预测摘要发送到该账号自己的飞书机器人。测试消息会明确标记为测试，不会写入归档；它用于验证第二天自动提醒依赖的真实数据、扫描逻辑和 Webhook 是否能跑通。

## 多个用户怎么区分数据

推荐方式是邀请制账号：

1. 管理员先创建邀请码。
2. 用户在页面右上角选择“邀请注册”。
3. 注册登录后，钱包列表、扫描结果、目标线和加成规则都会保存到自己的 Supabase 账号工作区。
4. 换设备时用同一个邮箱密码登录，就能恢复自己的归档。

这种方式比手填数据空间码更稳，不容易因为用户忘记空间码导致数据找不到。

旧版兼容方式如下：

`私有访问码` 只负责保护未登录旧路径的 API 额度，不负责区分用户数据。已经登录且账号状态正常的用户，可以直接使用受保护扫描接口，不需要再填写私有访问码。

真正区分数据的是钱包管理页里的 `数据空间码`：

- 同一个人换浏览器或换设备：填写同一个数据空间码。
- 不同用户：填写不同数据空间码。
- 默认空间是 `default`，会兼容旧版本已经保存的归档。

如果你新增了钱包地址，页面会把本地钱包列表和服务端钱包列表合并后再恢复，避免服务端旧归档把本地新增地址覆盖掉。钱包管理页也会显示服务端归档同步状态；如果访问码错误或同步失败，需要先修复这个状态，否则新地址不会写入服务端。

## 代币额外加成怎么填

额外加成不是全局代币规则，而是按下面三项绑定：

```text
钱包地址 + UTC 日期 + 代币地址
```

这样可以处理这些情况：

- 同一个代币 3 天前是 +50%，今天是 +20%。
- 不同钱包刷了同一个代币，但参与活动的日期不同。
- 某个代币只在某一天有额外加成。

填写加成后，页面会基于已有交易结果重新计算，不需要重新扫链。

## 数据来源怎么理解

页面会展示交易发现来源：

- `Ankr 索引`：最快，优先使用，用来按钱包找交易 hash。
- `Explorer 索引`：备选钱包交易索引；当前免费 Etherscan API 可能不支持 BSC。
- `RPC 兜底`：最后方案，通过 Chainstack/RPC 读取 Transfer 事件，速度较慢但可以兜底。
- `本地归档`：本地已有数据，刷新时没有必要重新扫链。

最理想配置是：

```text
Ankr：负责在 BNB Chain 和 X Layer 上按钱包找交易
Chainstack：负责解析 BNB Chain 交易和 BNB Chain RPC 兜底
X Layer RPC：负责解析 X Layer 交易；交易发现仍优先走 Ankr
```

## 当前边界

当前版本只统计：

```text
BNB Chain / X Layer 上的 OKX DEX Router 交易
```

当前版本暂不统计：

- Base
- Arbitrum
- Solana
- 其他链上的 OKX 钱包交易

如果你在 OKX 钱包里看到一笔 Boost 交易，但工具没算到，请把交易 hash 发给 Agent，让它先判断交易发生在哪条链上。

## 如果结果和 OKX 页面不完全一样

小幅差异是正常的。

这个工具主要根据链上稳定币侧成交额估算 USD；OKX 页面可能使用自己的历史价格、活动加成和内部口径。

如果差异很大，让 Agent 检查：

1. 快照日期是否选错。
2. 交易是否发生在其他链。
3. 代币额外加成是否漏填。
4. 这次是否走了 `RPC 兜底` 导致扫描较慢但还没完成。
5. 页面是否有未解析交易或警告。
