# S22 子 Plan：账号额度与扫描访问闭环加固

## 阶段目标

系统性审查账号、额度、归档和扫描访问链路，把已经暴露出来的“字段存在但不生效”“登录后仍依赖私有访问码”“长开页面登录态过期”等明显问题修到闭环。

## 实现边界

1. 钱包额度必须在服务端保存 Supabase 归档时强制校验，不能只在邀请码或前端列表里展示。
2. 登录且状态为 active 的 Supabase 用户，可以调用 `/api/rpc`、`/api/ankr`、`/api/explorer`，不再要求额外填写私有访问码。
3. 私有访问码仍可作为未登录旧路径的 API 保护方式；飞书 Webhook 仍保持全局访问码保护，避免普通用户直接消耗全局通知通道。
4. 管理员邀请码的默认额度按角色区分：普通用户默认 20 个钱包，管理员默认 200 个钱包。
5. 前端在账号超出钱包额度时停止自动同步和扫描动作，并展示可理解的额度反馈。
6. 页面长时间打开时自动刷新 Supabase session，避免扫描代理因为 access token 过期而失败。

## 验收标准

1. 未登录且无私有访问码时，扫描代理仍返回 401。
2. 登录 active 用户不填写私有访问码，也能调用 `/api/rpc`。
3. 普通用户超出钱包额度保存 `/api/archive` 时返回 403。
4. 普通用户未超额保存 `/api/archive` 时成功写入 Supabase。
5. 钱包管理页展示当前账号的钱包额度，超额时禁用刷新和重扫。
6. 测试用户、测试邀请码、测试工作区清理后，Supabase 不残留测试数据。

2. 将飞书 Webhook 下放为用户工作区级配置，避免所有用户共用一个全局机器人。
3. 增加管理员用户列表、禁用用户、调整钱包额度和查看用量的入口。
