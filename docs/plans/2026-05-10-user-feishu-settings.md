# S24 子 Plan：用户级飞书通知配置

## 阶段目标

把飞书提醒从全局环境变量升级为账号级配置。每个登录用户可以在偏好设置里保存自己的飞书机器人 Webhook；手动风险提醒和每日 Cron 自动提醒都优先使用该用户配置，避免多用户共用一个群机器人。

## 实现边界

1. Supabase `notification_settings` 增加 `feishu_secret`，支持飞书机器人签名。
2. `/api/auth` 新增用户动作：读取和更新当前账号的飞书通知配置。
3. 前端偏好设置页新增“飞书通知”卡片，支持保存 Webhook、签名密钥、启停提醒、预测未来天数和测试发送。
4. `/api/feishu` 对登录用户优先使用账号级 Webhook；未登录旧路径仍可用全局 `FEISHU_WEBHOOK_URL`。
5. 每日 Cron 对 Supabase 工作区优先读取工作区所属用户的通知配置；未配置时回退到全局 Webhook。
6. 返回给前端的 Webhook 和签名密钥必须脱敏，不返回原文。

## 验收标准

1. 登录用户可以读取默认飞书配置。
2. 登录用户可以保存官方飞书/Lark Webhook，并启用提醒。
3. 无效 Webhook 会被服务端拒绝。
4. 账号级配置不会把 Webhook 原文返回给前端。
5. 手动飞书提醒走登录用户配置。
6. Cron dry-run 可展示通知预览，真实发送前会按用户配置判断是否需要提醒。
7. 未配置账号级 Webhook 时，旧版全局 Webhook 仍可兜底。

## 当前状态

已完成。代码、Supabase 迁移、真实 API 测试、本地浏览器验证、生产部署和线上 smoke test 均已完成。

## 后续建议

1. 增加通知发送日志，记录每次提醒属于哪个工作区、是否发送成功。
2. 增加用户级每日刷新/重扫限额的真实扣减。
3. 管理员页面增加用户通知配置状态，只展示是否配置，不展示密钥。
