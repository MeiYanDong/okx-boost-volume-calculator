# S30 用量限制与规模边界审查

## 目标

继续从管理员、普通用户、滥用者、运维者和数据正确性五个视角审查明显错误，把“字段存在但未生效”“生效后无法配置”“多钱包数据被截断”“候选交易误收录”这类问题修到生产可用。

## 实现边界

1. `daily_refresh_limit` 和 `daily_rescan_limit` 必须真实限制登录用户的手动刷新与强制重扫。
2. 用量扣减不能使用会被并发覆盖的读后 upsert，改为带条件的乐观扣减。
3. 管理员创建邀请码和管理用户时，都能配置每日刷新上限和每日重扫上限。
4. 归档容量必须匹配当前 500 钱包上限，避免保存时静默截断钱包列表或记录。
5. Ankr 索引候选交易必须同时满足钱包 `from` 和 OKX router `to`，缺字段不再放行。

## 验收

- `npm run lint`
- `npm run build`
- 本地 API smoke：未登录扣减用量返回 401。
- 本地浏览器检查偏好设置页可看到每日刷新和每日重扫配置入口。
- 生产部署后验证首页、未授权 API、Cron 鉴权、auth me 和扣减用量接口。
