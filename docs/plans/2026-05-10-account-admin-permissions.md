# S21 子 Plan：账号级管理员权限

## 阶段目标

把邀请码管理从“谁知道私有访问码谁能管”升级为“登录 Supabase 管理员账号才能管”，同时保留首个管理员账号的安全初始化路径。

## 实现边界

1. `/api/auth?action=me` 返回用户角色、状态和钱包额度。
2. 登录和刷新 session 时，把 `app_profiles` 中的角色信息合并到前端 session。
3. 邀请码管理接口优先校验登录账号是否为 `admin`。
4. 当系统没有任何 active admin 时，允许用私有访问码创建第一个管理员邀请码。
5. 一旦存在 active admin，私有访问码不再允许管理邀请码；救援通道只保留服务端 `CRON_SECRET`。
6. 前端邀请码管理支持选择普通用户或管理员角色；登录 admin 后无需填写私有访问码。

## 验收标准

1. 无登录、无私有访问码时，邀请码管理接口返回 401。
2. 无 active admin 时，私有访问码创建的邀请码会被强制设为管理员邀请码。
3. 使用管理员邀请码注册后，session 中能拿到 `role=admin`。
4. 管理员登录态可以创建、列表、撤销邀请码。
5. 已有 active admin 后，单独使用私有访问码管理邀请码会返回 403。
6. 测试数据清理后，Supabase 不残留测试用户、profile、workspace 或测试邀请码。
