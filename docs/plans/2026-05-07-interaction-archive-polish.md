# S2 子 Plan：交互与归档完善

## 阶段目标

完善 S1 工作台布局后的交互细节，优先修复影响用户正常查看数据的问题。

## 当前状态

进行中。已完成详情抽屉滚动修复，将每日数据、代币额外加成、交易明细从长滚动信息堆叠重构为可独立滚动的 Tab 工作区，并修复单钱包重扫入口、加成规则粒度、单钱包增量刷新和批量钱包有限并行扫描。

## 范围

本阶段包含：

1. 详情抽屉滚动、关闭、查看体验。
2. 钱包列表编辑体验。
3. 钱包筛选与归档状态体验。
4. 不影响核心计算公式和链上扫描逻辑。

本阶段已经触及扫描调度逻辑，但只做归档层面的增量补扫，不改变 Boost 公式、代币倍数和交易解析口径。

## 已完成修复

详情抽屉原先使用固定层里的内部滚动，浏览器中滚轮/触控可能落到遮罩或背景层，导致详情无法滑动。

修复方式：

1. 明确让抽屉本体成为滚动容器。
2. 给抽屉增加 `100dvh` 高度、`overflow-y: auto`、移动端惯性滚动和纵向触控。
3. 让遮罩层不抢抽屉区域的指针事件。
4. 给抽屉增加可聚焦能力，支持键盘 PageDown 滚动。

详情内容原先只是三张基础表，用户很难快速判断每天贡献、代币加成和每笔交易状态。第一次修复后仍有问题，根因是详情页依赖长滚动和锚点导航；在 Codex in-app browser 的窄宽度下触发移动端断点后，摘要区、导航和详情区被纵向堆叠，核心内容被挤到首屏外，视觉验证也暴露每日数据区域信息不可读。

修复方式：

1. 详情页改为三段式 Tab 工作区，避免窄视口下依赖长滚动查找核心信息。
2. 摘要区改为紧凑 2x2/4 列响应式布局，避免移动端堆成四张大卡。
3. 每日数据改为 10 日贡献条，突出每日 Boost、成交额和有效笔数。
4. 代币额外加成改为代币卡片，展示代币、分组、关联交易数、当前加成和快捷档位。
5. 交易明细增加有效/部分/排除状态列，并保留哈希、日期、交易对、成交额、倍数和 Boost 计算结果。

第二次修复后仍有问题，根因是滚动职责仍放在详情抽屉外层；每日数据和交易明细内容虽然存在，但用户在内容区域滚动时没有明确命中可滚动容器。尤其交易明细外层原本主要处理横向滚动，纵向长表没有自己的滚动区域。

修复方式：

1. 详情抽屉改为固定高度、隐藏外层溢出，只保留顶部摘要与 Tab 控制。
2. `detail-tab-panel` 占用剩余高度。
3. 每日数据列表、代币加成列表、交易明细表格各自成为可聚焦、可触控、可键盘滚动的纵向滚动容器。
4. 交易明细表头固定，长表向下滚动时仍能看到列含义。

用户反馈单个钱包无法重新刷新，且同一代币在不同日期、不同钱包的加成可能不同。根因是：

1. 单钱包强制重扫只做成了图标按钮，且在横向表格的最右侧，窄屏下不可发现。
2. 代币加成原先以 `token -> bonus` 存储，是全局规则，会影响所有钱包、所有日期的同一个代币。

修复方式：

1. 钱包列表操作列固定在表格右侧，单钱包重扫改为明确文字按钮 `重扫`。
2. 钱包详情顶部增加 `重扫该钱包` 操作。
3. 加成规则存储格式升级为 `钱包地址|UTC日期|代币地址=加成百分比`。
4. 代币加成 Tab 按 `日期 + 代币 + 钱包` 生成配置项，同一代币不同日期会拆成不同项。
5. 重算 Boost 时只应用匹配当前钱包、当前交易日期、当前交易代币的加成；历史全局 token 规则仅作为兼容 fallback。

用户继续追问“一个钱包如果有新的交易，如何只计算新的交易就行了”。根因是：

1. 归档结果只保存交易结果，没有保存本次扫描到的区块高度。
2. 普通刷新逻辑命中归档后直接返回，不会检查归档之后的新区块。
3. 单钱包只有强制重扫入口，缺少“补扫新增交易”的主路径。

修复方式：

1. 计算结果增加 `scannedFromBlock`、`scannedToBlock`、`incrementalFromBlock`、`incrementalNewTxCount`。
2. 普通刷新携带上一份归档结果，只从上次扫描区块附近开始补扫，并保留 200 个区块的安全重叠。
3. 增量刷新只解析未见过的新 hash，再和旧 swaps 合并后重算 10 日汇总。
4. 钱包列表对已归档钱包展示 `刷新` 和 `重扫` 两个动作；详情页展示 `刷新该钱包` 和 `重扫`。
5. 旧归档如果没有区块检查点，首次刷新会完整补建一次检查点，之后才能走真正增量。

用户进一步要求“改良为最优版本，使用 ANKR 或者其他方法”。根因是上一版虽然已经增量补扫，但仍有两处浪费：

1. 已有归档覆盖最新区块时，仍可能继续进入候选交易发现流程。
2. Ankr 钱包索引翻页虽然按区块过滤，但遇到低于增量起点的记录时没有立刻停止。
3. 用户看不到本次到底是 Ankr、Explorer 还是 RPC 兜底。

修复方式：

1. 刷新时先比较 `endBlock` 和上一份归档的 `scannedToBlock`，如果已经覆盖最新区块，直接返回归档结果，不读取 Transfer 事件。
2. Ankr Advanced 钱包索引优先；增量查询遇到低于 `scanStartBlock` 的交易立即停止翻页。
3. Explorer 钱包交易索引作为第二优先级。
4. RPC Transfer 事件只作为最后兜底，并在进度文案中明确标记为 `RPC 兜底`。
5. 计算结果记录 `txDiscoverySource`，前端展示 `Ankr 索引`、`Explorer 索引`、`RPC 兜底` 或 `本地归档`。

用户提出“可以实现并行扫”。根因是批量刷新原先按钱包串行执行，即使单钱包内部已经有 RPC 并发，多钱包场景仍然会被串行队列拖慢。

修复方式：

1. 批量刷新和批量强制重扫改为 3 路有限并发队列。
2. 单钱包刷新仍保持独立状态、独立错误和独立进度。
3. 并发上限不做过高，避免 Ankr、Explorer 或 RPC 兜底时被请求放大打爆。
4. 批量完成后汇总成功/失败数量。

## 验收标准

1. 打开钱包详情后可以纵向滚动。
2. 每日数据、代币额外加成、交易明细都能继续访问。
3. 详情关闭按钮仍可用。
4. `npm run lint` 通过。
5. `npm run build` 通过。
6. 浏览器验证每日数据为 10 行。
7. 浏览器验证代币加成输入和快捷档位可见。
8. 浏览器验证交易明细包含交易状态和 BscScan 哈希链接。
9. 窄视口截图验证每日数据文本、金额和进度条均可见。
10. 验证每日数据容器 `scrollTop` 可从 0 滚到末尾。
11. 验证交易明细容器 `scrollTop` 可从 0 滚到末尾并看到最后一笔交易。
12. 钱包列表可直接看到单钱包 `重扫` 入口。
13. 同一代币设置某一天加成后，不影响该钱包其他日期的同一代币交易。
14. 钱包列表可直接看到单钱包 `刷新` 入口。
15. 普通刷新命中已有归档时不会清空当前详情数据。
16. 新结果写入本地归档后保存扫描到的最新区块。
17. 增量刷新完成后能看到新增 OKX 交易数。
18. 已归档覆盖最新区块时，刷新不会读取 Transfer 事件。
19. Ankr 索引低于增量起点后停止翻页。
20. 前端能看到本次交易发现来源。
21. 批量扫描以有限并发执行。
22. 单个钱包失败不阻塞其他钱包继续扫描。
