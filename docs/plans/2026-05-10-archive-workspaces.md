# S17 子 Plan：归档数据空间隔离

## 阶段目标

修复服务端归档的两个数据可靠性问题：

1. 不同用户使用同一个私人部署时，不能共享同一份 Upstash 归档。
2. 用户新增钱包地址后，后续打开页面不能被服务端旧归档覆盖导致地址消失。

## 实现边界

1. 不引入登录系统，继续保持私人部署的轻量形态。
2. `ACCESS_PASSWORD` 仍只负责保护 API 和额度，不再承担用户数据隔离。
3. 新增“数据空间码”作为归档命名空间：同一用户换设备填写同一个码，不同用户填写不同码。
4. Vercel Cron 默认遍历所有已登记的数据空间；也允许通过 `workspace` 参数单独刷新某个数据空间。
5. 兼容旧的全局归档 key，默认数据空间仍可读取历史数据。

## 实现方案

1. Upstash 归档 key 从单一 `okx-boost:server-archive:v1` 改为按数据空间 hash 后的 v2 key。
2. 维护数据空间索引，保存所有已写入过的空间码。
3. `/api/archive` 支持 `x-okx-boost-workspace` header、`workspace` query 和 body `workspaceId`。
4. 前端钱包管理页新增“数据空间码”输入。
5. 前端启动恢复服务端归档时合并本地钱包列表和服务端钱包列表，避免新增地址丢失。
6. 前端展示服务端归档同步状态，避免同步失败完全无感。

## 当前状态

已完成本地实现，待完成构建验证和生产部署。

## 验收标准

1. 不同数据空间码写入不同 Upstash key。
2. 默认数据空间可读取旧全局归档。
3. 新增钱包地址后，服务端旧归档不会覆盖本地新增地址。
4. Cron 不指定空间时会刷新所有已登记空间。
5. `npm run build` 通过。
6. 生产部署完成。
